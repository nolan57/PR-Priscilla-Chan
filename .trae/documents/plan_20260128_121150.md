## Fix Dtype Mismatch in validate_npz.py

### Problem Analysis
The error occurs because the `extract_ground_truth_f0` function returns inconsistent dtypes:
- Line 82: When using external F0, librosa.load() returns float32 (default)
- Line 87: When falling back to parselmouth, explicitly returns float64

This inconsistency causes the dtype mismatch when the waveform is passed to `get_mel_torch`, which creates a tensor with the same dtype, conflicting with the float32 mel basis.

### Solution
Modify the `extract_ground_truth_f0` function in validate_npz.py to ensure consistent dtype (float32) for the returned waveform, regardless of which path is taken.

### Implementation Steps
1. Edit validate_npz.py line 87 to remove the explicit float64 dtype specification
2. Change from: `y, sr = librosa.load(wav_path, sr=SAMPLE_RATE, mono=True, dtype=np.float64)`
3. Change to: `y, sr = librosa.load(wav_path, sr=SAMPLE_RATE, mono=True)`

This ensures both code paths return float32 arrays, which will create float32 tensors in `get_mel_torch`, matching the mel basis dtype and resolving the mismatch error.
